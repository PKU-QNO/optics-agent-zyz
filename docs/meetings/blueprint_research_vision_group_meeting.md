# 蓝图化科研自动化组会笔记

更新时间：2026-06-10

## 核心判断

我们现在做的不是“只复刻几篇光学论文”，而是在用论文复刻训练一套可迁移的科学计算蓝图框架。

更准确的目标链条是：

```text
复刻论文 X
  -> 抽象出可复用蓝图
  -> 蓝图成为通用科研自动化框架
  -> 蓝图辅助新的科学发现
```

论文复刻只是起点，不是终点。光学也只是当前经费和切入方向，不应该把系统能力限制在光学里。真正要做的是：让 AI 能进入任何它原本不熟悉的科学/工程领域，通过文献、理论、仿真、扫参和验证，逐步形成可执行的科研工作流。

## 要说清楚的定位

当前项目可以这样理解：

```text
论文复现 = 蓝图迭代的训练样本
光学方向 = 当前应用入口和经费来源
COMSOL/Magnus = 科学计算执行后端
蓝图 = 可迁移的科研自动化协议
DSL = 未来让蓝图可组合、可搜索、可生成的语言层
```

所以我们不是只做一个“光学仿真脚本库”，而是做一个“AI 驱动的科学问题求解系统”。它先在光学里跑通，再迁移到流体、材料、生物物理、等离子体、工程优化等领域。

## 目标层次

### 第一层：复刻论文 X

这是最具体、最容易验收的一层。

输入：

```text
一篇论文
```

输出：

```text
理论理解
模型假设
参数表
数值程序
仿真结果
和论文图表/指标的对齐
```

这一步的价值不是“复现这篇论文本身”，而是逼迫系统面对真实论文里的不完整描述、隐含参数、单位换算、边界条件、数值稳定性和工程细节。

### 第二层：蓝图用于通用框架

当多个论文复刻完成后，要抽象出重复结构：

```text
读论文
提取物理/数学模型
整理参数
生成 case
提交仿真
扫参
分析结果
对齐论文
记录失败原因
迭代修正
```

这些重复结构应该沉淀成蓝图，而不是每篇论文重新写一套流程。

蓝图应该逐渐具备：

- 明确输入
- 明确输出
- 明确资源需求
- 明确失败模式
- 明确可扫参数
- 明确结果验证方式
- 明确可复用的 runner 和后处理接口

### 第三层：蓝图用于新的科学发现

当蓝图不仅能复刻论文，还能稳定扫参、比较结果、发现异常模式、提出下一轮参数组合时，它就不只是复现工具，而是科研探索工具。

这时目标会变成：

```text
已知论文结论
  -> 建立可运行模型
  -> 扫描更大参数空间
  -> 发现论文没覆盖的区域
  -> 找到新现象/新结构/新优化点
```

也就是说，复刻论文是蓝图的校准过程；新的科学发现才是更高一级目标。

## 三个重点方向

### 1. 扫参：把科研问题变成可探索空间

扫参不只是改几个物理参数。应该分成几类：

```text
数学参数
物理参数
几何参数
材料参数
边界条件
solver 参数
网格参数
算力参数
后处理指标
```

例如 COMSOL 任务里，扫参可以包括：

- 波长、频率、入射角
- 折射率、黏度、密度、热导率
- 几何尺寸、间距、曲率、孔径
- 网格密度、收敛阈值、时间步长
- CPU 数、内存、任务优先级、并行粒度
- 结果指标，例如透过率、Q factor、压力降、涡量、模式频率

通用蓝图需要把这些参数统一描述出来，而不是散落在 `.java`、`.mph`、shell 脚本和人工笔记里。

一个理想的 case 描述应该接近：

```text
model:
  domain: optics | fluid | generic
  equations: ...
parameters:
  physical: ...
  geometric: ...
  numerical: ...
  compute: ...
sweep:
  variables: ...
  strategy: grid | random | bayesian | adaptive
metrics:
  primary: ...
  constraints: ...
outputs:
  tables: ...
  figures: ...
  manifest: ...
```

### 2. 单个完备蓝图完成多个论文复现，甚至创新

蓝图不能只服务一篇论文。真正有价值的蓝图应该能覆盖一类论文。

例如一个成熟的光学仿真蓝图，不应该只是：

```text
复现论文 A 的图 2
```

而应该是：

```text
处理一类 waveguide / metasurface / cavity / scattering 问题
```

输入不同 case 后，可以完成：

- 论文 A 的复现
- 论文 B 的复现
- 论文 C 的复现
- 对某个参数空间的扩展搜索
- 对论文之外结构的优化

这要求蓝图内部不要写死某篇论文的特殊逻辑，而要把论文差异放进 case/config 里。

最终希望形成：

```text
一个完备蓝图
  + 多个 case
  + 多个后处理指标
  + 参数搜索策略
  = 多篇论文复现和新结构探索
```

### 3. DSL 工作：让蓝图可生成、可组合、可检查

后续需要 DSL，因为只靠自然语言和脚本会越来越乱。

DSL 的作用不是为了“造一门语言”本身，而是为了让 AI 和人都能稳定表达科研任务：

```text
我要解什么方程
在哪个几何里解
边界条件是什么
材料参数是什么
扫哪些变量
用什么 solver
看什么指标
怎么算成功
失败后怎么调整
```

DSL 可以先从简单的 JSON/YAML schema 开始，不必一开始就复杂。

第一阶段 DSL 可以服务于：

- case 描述
- 参数表
- sweep 计划
- 输出指标
- 资源需求
- 失败分类

后续再扩展到：

- 方程模板
- 几何模板
- solver 策略
- 主动学习/贝叶斯优化
- 跨仿真软件后端

## 和 COMSOL 镜像的关系

COMSOL 镜像现在的意义是：我们终于有了一个能在 Magnus 上执行科学仿真的后端。

当前 active image：

```text
docker://magnus-local/comsol-runtime:latest
```

它已经通过：

- COMSOL 版本检查
- batch 模式检查
- Python 后处理依赖检查
- license mount 后实际求解
- L-shaped membrane eigenmodes 示例

这说明执行后端这条路已经打通。接下来重点不是继续折腾镜像，而是把它接入蓝图、case、DSL 和扫参流程。

换句话说：

```text
COMSOL runtime = 执行引擎
Magnus blueprint = 调度协议
case/DSL = 科学问题表达
runner = 执行和结果收集
manifest = 机器可读的实验记录
```

## 下一步落地计划

### Step 1：正式蓝图 env_check

用当前 active image 跑一次正式蓝图：

```powershell
python comsol\automation\submit_comsol.py --run-mode env_check --license-mode personal_storage --container-image docker://magnus-local/comsol-runtime:latest
```

目标是确认：

- 蓝图默认镜像正确
- runner 路径正确
- license 路径正确
- 输出目录可写
- `manifest.json` 正常生成

### Step 2：光学最小 smoke case

做一个最小光学 case，例如：

- 2D waveguide
- Helmholtz frequency-domain
- 简单 cavity/mode 求解

目标不是追求复杂，而是验证：

- 光学相关模块或等效 PDE 表达能跑
- 参数能进入模型
- 结果能被后处理读取
- manifest 能记录关键指标

### Step 3：流体最小 smoke case

做一个最小流体 case，例如：

- laminar channel flow
- microfluidics pressure-driven flow
- 简单 Stokes/Navier-Stokes 测试

目标是确认：

- 流体方向可用
- license 覆盖情况清楚
- solver 能稳定收敛
- 后处理指标能输出

### Step 4：定义 case DSL v0

先不要追求完美 DSL，先定义一个能跑的 v0：

```text
case.json
parameters.json
sweep.json
metrics.json
```

或者合并为一个：

```text
case.yaml
```

核心目标是让每次仿真都能被机器读懂，而不是只靠人看脚本。

### Step 5：从单论文复刻扩展到多论文复刻

选一类问题，比如 waveguide 或 membrane/eigenmode，尝试让同一个蓝图处理多篇论文/多个 case。

验收标准：

```text
不是跑通一篇论文
而是同一套蓝图跑通一类论文
```

## 组会表达建议

可以这样跟组里说：

> 我们当前不是把目标限制在“复刻一篇光学论文”。复刻论文只是训练蓝图的起点。真正目标是做一个 AI 可以迁移到不同科学领域的科研自动化框架。光学是当前经费来源和应用入口，COMSOL/Magnus 是计算后端，蓝图和 DSL 是我们要沉淀的通用能力。

也可以这样说：

> 下一阶段重点应该从“有没有镜像”转到“怎么把科学问题表达成可扫参、可复现、可迭代的蓝图”。我们希望一个完备蓝图可以服务多篇论文，甚至在复现之后继续探索论文没有覆盖的参数空间，从而支持新的发现。

## 当前最重要的原则

1. 不要动管理员已经导入的 active image。
2. 不要把项目做成单篇论文脚本集合。
3. 每次复现都要抽象出可复用蓝图能力。
4. 扫参和 DSL 是从复现走向发现的关键。
5. 光学是入口，不是边界。
